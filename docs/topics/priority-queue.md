# PriorityQueue

## 1. 概要

`PriorityQueue` は、優先度の高い要素を先に取り出せる Java 標準ライブラリのキュー。  
内部ではヒープを使っており、通常は**最小値が先に取り出される**。

今回のトピックでは、次の内容を確認する。

- 最小優先キュー
- 最大優先キュー
- 追加
- 先頭確認
- 取り出し順
- `PriorityQueue` とヒープの関係

## 2. 基本の動き

`PriorityQueue` では、追加順ではなく優先度順で先頭が決まる。  
整数をそのまま入れる場合、既定では小さい値が先に出てくる。

このトピックでは、次の流れで `PriorityQueue` を確認する。

1. 既定の `PriorityQueue` へ値を追加する
2. 先頭には最小値が来ることを確認する
3. 取り出し順が昇順になることを確認する
4. `Collections.reverseOrder()` を使って最大優先キューも作る
5. 取り出し順が降順になることを確認する

## 3. 処理の流れ

このトピックでは、次の順で `PriorityQueue` の状態を確認する。

1. 入力値を最小優先キューへ 1 つずつ追加する
2. 追加後の並びを学習用の見やすい形で確認する
3. `peek()` で先頭値を確認する
4. `poll()` を繰り返し、取り出し順を確認する
5. 同じ入力値を最大優先キューへ追加する
6. 最大優先キューの `peek()` と取り出し順も確認する

trace あり実行では、追加時の先頭値や取り出し値を確認できる。  
そのため、最小優先キューと最大優先キューで何が違うかを出力から追える。

## 4. 計算量

### 追加

- 時間計算量: O(log n)
- 空間計算量: O(1)

### 先頭確認

- 時間計算量: O(1)
- 空間計算量: O(1)

### 取り出し

- 時間計算量: O(log n)
- 空間計算量: O(1)

`PriorityQueue` もヒープを使っているため、追加と取り出しは木の高さぶんだけ比較が発生する。  
そのため、計算量はヒープと同じ考え方になる。

## 5. メリット / デメリット

### メリット

- 標準ライブラリで使える
- 最小値や最大値を効率よく取り出せる
- 自前ヒープを実装しなくても優先度付き処理を扱える

### デメリット

- iterator の順は優先順ではないため、そのままでは中身が見づらい
- 配列や通常のキューより挙動が直感的でないことがある
- 内部構造は隠れているため、学習用には自前ヒープと合わせて見る方が理解しやすい

## 6. Java実装のポイント

- 既定の `PriorityQueue<Integer>` は最小値が先に出る
- `Collections.reverseOrder()` を渡すと、最大値が先に出る PriorityQueue を作れる
- 学習用に、追加後の内容は見やすい並びへ並べ替えて表示している
- `peek()` で先頭値を確認し、`poll()` で取り出し順を確認する
- runner 側では入力決定と結果表示を担当し、学習本体は `algorithms/` に寄せる

## 7. コアとなる処理・重要なコードの見どころ

`PriorityQueue` で重要なのは、**追加順ではなく優先順で先頭が決まる点** と、**比較器で取り出し順を変えられる点**。

### 最小優先キュー

既定の `PriorityQueue` では、小さい値が先頭になる。

```java
PriorityQueue<Integer> minQueue = new PriorityQueue<>();
```

このキューへ値を追加して `poll()` を繰り返すと、値は小さい順に出てくる。

### 最大優先キュー

比較器を渡すと、優先順位のルールを変えられる。

```java
PriorityQueue<Integer> maxQueue = new PriorityQueue<>(Collections.reverseOrder());
```

この設定により、大きい値が先に出るキューとして使える。  
ここで見たいのは、ヒープ構造そのものを自前実装しなくても、標準ライブラリで同じ考え方を使える点。
